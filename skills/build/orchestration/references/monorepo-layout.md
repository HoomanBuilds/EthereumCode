# Monorepo Layout: Scaffold-ETH 2

The directory structure SE2 ships with is opinionated and load-bearing вҖ” agents that "rearrange to fit" break the build pipeline. This file maps every directory, what auto-generates vs what you author, and where to put new code.

For deploy mechanics that depend on this layout, see `frontend-playbook/SKILL.md`. For why these phases exist, see `references/phase-gates.md`.

## Top-level

```
my-dapp/
в”ңв”Җв”Җ package.json           вҶҗ root workspace package
в”ңв”Җв”Җ yarn.lock
в”ңв”Җв”Җ .yarnrc.yml            вҶҗ yarn berry config (zero-installs off by default)
в”ңв”Җв”Җ .gitignore
в”ңв”Җв”Җ README.md
в””в”Җв”Җ packages/
    в”ңв”Җв”Җ foundry/           вҶҗ Solidity + tests + scripts
    в””в”Җв”Җ nextjs/            вҶҗ Next.js frontend
```

Yarn workspaces. The root `package.json` orchestrates scripts across packages: `yarn deploy` вҶ’ calls into `packages/foundry`, `yarn start` вҶ’ calls into `packages/nextjs`.

**Don't add a third workspace** unless you have a clear non-frontend, non-contract concern (subgraph indexer, off-chain relayer service). Even then, prefer a separate repo unless tight coupling is justified.


### What you author

- `contracts/*.sol` вҖ” your contracts.
- `script/*.s.sol` вҖ” deploy scripts. Each is a `forge script` runnable.
- `test/*.t.sol` вҖ” tests. Mirror the `contracts/` structure.
- `foundry.toml` вҖ” solc version, optimizer runs, remappings.

### What auto-generates

- `deployments/<chainId>/` вҖ” written by SE2's deploy hook after `yarn deploy`. Contains contract address + ABI. Read by the frontend's contract generator.
- `broadcast/` вҖ” Foundry's transaction broadcast logs. **Always gitignored** вҖ” contains RPC URLs and tx hashes that may leak deployer info.
- `lib/` вҖ” managed by `forge install` / git submodules. Don't hand-edit; commit the submodule pointer.

### `foundry.toml` baseline

```toml
[profile.default]
src = "contracts"
out = "out"
libs = ["lib"]
test = "test"
script = "script"
solc_version = "0.8.28"
optimizer = true
optimizer_runs = 200
via_ir = false                  # turn on for hot inner loops
remappings = [
  "@openzeppelin/=lib/openzeppelin-contracts/",
  "forge-std/=lib/forge-std/src/",
]

[profile.deep]                   # for deep fuzz / slow CI
fuzz = { runs = 10000 }
invariant = { runs = 256, depth = 100 }
```

### What you author

- `app/` вҖ” pages. Use App Router conventions. Mark client components with `"use client"`.
- `components/` вҖ” your components. Don't edit `components/scaffold-eth/` in place; copy and rename if you need to fork them.
- `contracts/externalContracts.ts` вҖ” addresses + ABIs of OTHER protocols you call (Uniswap, Aave, USDC). Edit BEFORE Phase 1.2.
- `scaffold.config.ts` вҖ” main config: target networks, RPC overrides, polling, wallet options.
- `public/` вҖ” static assets. Custom thumbnail, favicon, manifest.

### What auto-generates

- `contracts/deployedContracts.ts` вҖ” written after every `yarn deploy`. Contains addresses + ABIs of YOUR deployed contracts. **Never hand-edit.** Edits get overwritten.
- `next-env.d.ts` вҖ” Next's type declarations.
- `.next/` вҖ” build cache; gitignore.
- `out/` вҖ” IPFS export output; gitignore.

### `scaffold.config.ts` essentials

```ts
import * as chains from "viem/chains";

const config = {
  targetNetworks: [chains.base],          // production target
  pollingInterval: 3000,                   // 3s feels responsive
  alchemyApiKey: process.env.NEXT_PUBLIC_ALCHEMY_API_KEY ?? "",
  rpcOverrides: {
    [chains.base.id]: process.env.NEXT_PUBLIC_BASE_RPC,
    [chains.mainnet.id]: process.env.NEXT_PUBLIC_MAINNET_RPC,
  },
  walletConnectProjectId: process.env.NEXT_PUBLIC_WC_PROJECT_ID ?? "",
  onlyLocalBurnerWallet: true,             // CRITICAL: hide burner on prod
  walletAutoConnect: true,
} as const;
```

**Never** put a literal API key here. Always `process.env.NEXT_PUBLIC_*`.

## externalContracts.ts pattern

```ts
const externalContracts = {
  8453: {                                  // Base mainnet
    USDC: {
      address: "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
      abi: [...],                          // copy from BaseScan
    },
    UniswapV3Router: {
      address: "0x2626664c2603336E57B271c5C0b26F421741e481",
      abi: [...],
    },
  },
  31337: {                                 // local fork
    USDC: { address: "0x833...", abi: [...] },  // SAME address on fork
    UniswapV3Router: { address: "0x2626...", abi: [...] },
  },
} as const;
```

Then in components:

```tsx
const { data: balance } = useScaffoldReadContract({
  contractName: "USDC",
  functionName: "balanceOf",
  args: [address],
});
```

The hook resolves contractName against deployedContracts (yours) + externalContracts (theirs).

## Path conventions

| Path | What |
|---|---|
| `~/components/...` | tsconfig path alias to `packages/nextjs/components` |
| `~~/components/...` | DOUBLE-tilde alias (some SE2 setups use this for nextjs/) |
| `@/...` | NOT used in SE2 вҖ” don't add it; conflicts with patterns |

Check your `tsconfig.json` paths section before importing вҖ” SE2 v0.x and v1.x differ.

## Where new code goes

| Adding... | Location |
|---|---|
| New contract | `packages/foundry/contracts/<Name>.sol` |
| New deploy step | `packages/foundry/script/Deploy<Name>.s.sol`, register in `Deploy.s.sol` |
| New page | `packages/nextjs/app/<route>/page.tsx` |
| New component | `packages/nextjs/components/<Name>.tsx` |
| External protocol | `packages/nextjs/contracts/externalContracts.ts` |
| Custom hook | `packages/nextjs/hooks/<name>.ts` (NOT under scaffold-eth/) |
| Subgraph types | `packages/nextjs/services/subgraph/` (or `packages/subgraph/` if heavy) |
| Off-chain API | New workspace `packages/api/` (or separate repo) |

## Common rearrangements that break SE2

- **Moving `contracts/` to `src/`**: SE2's contract generator hardcodes `contracts/`. Rewrite the hook or don't.
- **Renaming `packages/foundry` to `packages/contracts`**: yarn scripts in root `package.json` reference `packages/foundry` by name. Update both.
- **Adding `tsconfig.base.json`**: SE2 doesn't expect a shared base. Either commit fully (paths + references) or skip.
- **Pre-installing dependencies in CI before yarn workspaces**: Vercel and other CI need `cd ../.. && yarn install` from the package root, not pre-installed flat node_modules.
- **Replacing `yarn` with `pnpm` or `npm`**: SE2's scripts use yarn. Some break on pnpm hoisting. Audit before switching.

## Multi-chain layout

If you target multiple chains:

```
packages/foundry/script/
в”ңв”Җв”Җ Deploy.s.sol              вҶҗ reads $TARGET_CHAIN env
в”ңв”Җв”Җ chains/
в”Ӯ   в”ңв”Җв”Җ BaseDeploy.s.sol
в”Ӯ   в”ңв”Җв”Җ ArbitrumDeploy.s.sol
в”Ӯ   в””в”Җв”Җ MainnetDeploy.s.sol
в””в”Җв”Җ ...

packages/nextjs/scaffold.config.ts
  вҶ’ targetNetworks: [chains.base, chains.arbitrum, chains.mainnet]

packages/nextjs/contracts/externalContracts.ts
  вҶ’ keyed by chainId (8453, 42161, 1)
```

The frontend uses `useChainId()` to fetch the right addresses. Test the chain-switch flow end-to-end.

## Common pitfalls

- **Editing `deployedContracts.ts` by hand** вҶ’ next `yarn deploy` overwrites your changes. Put external contracts in `externalContracts.ts` instead.
- **Forgetting to commit `deployedContracts.ts`** for the prod chain вҶ’ CI rebuilds the frontend with stale addresses вҶ’ users send funds to nowhere. Either commit it or have CI run `yarn deploy --network <prod>` before frontend build.
- **`.env` not in `.gitignore`** вҶ’ leaks RPC keys, deployer keys.
- **`broadcast/` committed** вҶ’ leaks deployer address + RPC URL.
- **`scaffold-eth` components edited in place** вҶ’ SE2 update overwrites your changes. Copy + rename when forking.
- **Mixing two SE2 versions across packages** вҶ’ frontend hooks expect `deployedContracts.ts` shape that backend doesn't generate.
- **Foundry profiles colliding with CI flags** вҶ’ CI sets `FOUNDRY_PROFILE=ci`, your local `.env` overrides вҶ’ tests run different config locally vs CI.
