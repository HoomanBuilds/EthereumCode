# Eligibility Checklist

Before writing a single word of your grant proposal, verify eligibility. Most rejections happen because the applicant doesn't meet basic requirements.

## Universal Requirements

These apply to nearly all grant programs:

- [ ] **Open-source license.** Your code must be MIT, GPL, Apache-2.0, or similar. Closed-source = automatic rejection for most programs.
- [ ] **Deployed on the target chain.** If applying to Arbitrum, your contracts must be deployed on Arbitrum (or testnet with deployment plan). You can't apply to build on a chain you haven't touched.
- [ ] **Working prototype.** Most programs don't fund "idea stage." Have something running.
- [ ] **Public GitHub repo.** Reviewers will look at your code. Private repos = rejection.
- [ ] **Team identification.** Many programs require KYC or at least pseudonymous identities with track records.
- [ ] **No active exploits or unresolved security issues.** If your project has a known vulnerability, fix it first.

## Program-Specific Checks

### Optimism RPGF
- [ ] Project has demonstrable impact on the OP ecosystem
- [ ] Active in the ecosystem for at least 3 months
- [ ] Has an attestation or badgeholder endorsement
- [ ] Not a duplicate submission from a prior round (unless significantly expanded)

### Arbitrum Foundation
- [ ] Deployed on Arbitrum One or Nova (testnet acceptable for early rounds)
- [ ] Clear plan for driving TVL or transaction volume
- [ ] Not a fork of an existing project without meaningful differentiation
- [ ] Team has shipped at least one project before (or has a technical advisor who has)

### Ethereum Foundation
- [ ] Work is protocol-level or ecosystem-wide (not chain-specific)
- [ ] No commercial token or revenue model (or revenue model is secondary)
- [ ] Technical depth matches EF standards (research-grade or production tooling)
- [ ] Not already funded by another EF grant for the same work

### Gitcoin
- [ ] Project is a public good (free or open-source)
- [ ] Has a clear use case and existing users or a strong waitlist
- [ ] Not a commercial product disguised as a public good
- [ ] Team can commit to the round timeline (typically 4-8 weeks)

### L2 Ecosystem Funds (Base, Polygon, etc.)
- [ ] Native to the chain (not multi-chain with the target chain as an afterthought)
- [ ] Aligns with the chain's stated ecosystem priorities
- [ ] Has a path to user acquisition on that chain

## Red Flags That Get You Rejected Immediately

| Red Flag | Why it's a problem |
|---|---|
| Fork with no changes | No ecosystem benefit; the original already exists |
| Team has zero shipped code | Too risky for the program's budget |
| Token launch as the primary goal | Grants are for builders, not token fundraisers |
| "We'll figure out the details later" | Reviewers need concrete milestones |
| Copy-pasted proposal across programs | Reviewers talk; they notice |
| Budget includes marketing spend | Most programs fund development, not marketing |
| Prior grant was not delivered | You're blacklisted until you complete the prior work |

## Self-Assessment Score

Before applying, rate yourself:

| Criteria | Score (1-5) |
|---|---|
| Prior shipped work relevant to this project | /5 |
| Active users or clear demand signal | /5 |
| Technical depth of the team | /5 |
| Alignment with program priorities | /5 |
| Realistic timeline and budget | /5 |

**20-25:** Strong candidate. Apply.
**15-19:** Viable but risky. Consider building more before applying.
**Below 15:** Not ready. Focus on shipping, not fundraising.

## What to Do If You're Not Eligible

1. **Ship something.** A small, working tool on the target chain is worth more than a proposal.
2. **Contribute to existing projects.** Open-source contributions build your track record.
3. **Apply for a smaller grant first.** Build credibility, then apply for larger programs.
4. **Partner with an experienced team.** If you have the idea but not the track record, find someone who has shipped before.
